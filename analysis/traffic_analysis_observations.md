## 2. Observation 1 – DNS Traffic

### Filter used
```
dns
```

### What was observed

Multiple DNS query and response pairs were visible:

**Example DNS Query packet:**
```
Source:       192.168.1.105 (my machine)
Destination:  192.168.1.1   (home router / DNS resolver)
Protocol:     DNS
Info:         Standard query 0x1a2b A google.com
```

**Example DNS Response packet:**
```
Source:       192.168.1.1
Destination:  192.168.1.105
Protocol:     DNS
Info:         Standard query response 0x1a2b A google.com → 142.250.195.46
```

### Key observations
- DNS queries are sent in **plaintext UDP on port 53** — anyone on the network can see which domains you are resolving
- The query-response round trip took approximately **12ms**
- Multiple domains were queried automatically by the browser even before any page fully loaded (CDN domains, analytics, fonts)
- DNS traffic reveals browsing intent even when HTTPS encrypts the actual content

### Security implication
DNS over plain UDP is a significant privacy leak. An attacker on the same network performing ARP poisoning could:
1. See every domain the victim resolves
2. Respond with fake DNS answers (DNS spoofing) redirecting the victim

### Mitigation observed
Using **DNS over HTTPS (DoH)** or **DNS over TLS (DoT)** encrypts DNS queries, preventing eavesdropping on domain lookups.

## 3. Observation 2 – HTTPS (TLS) Traffic

### Filter used
```
tcp.port == 443
```

### What was observed

Large volume of HTTPS traffic to various IPs. Selected one TCP stream from visiting `https://github.com`:

**TLS Handshake visible:**
```
Packet 1:  Client Hello     → TLS 1.3, cipher suites offered
Packet 2:  Server Hello     → TLS 1.3 selected, certificate sent
Packet 3:  Certificate      → github.com certificate (RSA 2048)
Packet 4:  Client Finished  → Encrypted handshake complete
Packet 5+: Application Data → [ENCRYPTED — content not visible]
```

**Application data packets:**
```
Source:      192.168.1.105
Destination: 140.82.121.4 (GitHub)
Protocol:    TLSv1.3
Info:        Application Data  (encrypted payload)
Length:      varies (1400 bytes per packet)
```

### Key observations
- After the TLS handshake, **all data is encrypted** — Wireshark shows `Application Data` but cannot decrypt the content
- The **TLS 1.3 handshake** is faster than TLS 1.2 — only 1 round trip needed
- **SNI (Server Name Indication)** field in the Client Hello reveals the hostname being visited — even over HTTPS. This is visible in plaintext in the TLS handshake.
```
Extension: server_name
  Server Name: github.com      ← visible in plaintext!
```
- This means even HTTPS traffic reveals which domain you are visiting (though not the specific URL or content)

### Security implication
HTTPS protects content but the destination domain is visible via SNI in the TLS handshake. Newer standard **Encrypted Client Hello (ECH)** addresses this by encrypting the SNI field.

## 4. Observation 3 – ICMP (Ping) Traffic

### Filter used
```
icmp
```

### Command run
```bash
ping google.com -n 4     # Windows
ping google.com -c 4     # Linux
```

### What was observed

```
Packet 1:  Echo (ping) request   192.168.1.105 → 142.250.195.46   64 bytes
Packet 2:  Echo (ping) reply     142.250.195.46 → 192.168.1.105   64 bytes  TTL=118  time=15ms
Packet 3:  Echo (ping) request   192.168.1.105 → 142.250.195.46   64 bytes
Packet 4:  Echo (ping) reply     142.250.195.46 → 192.168.1.105   64 bytes  TTL=118  time=13ms
...
```

**Packet details for one ICMP request:**
```
Internet Control Message Protocol
  Type: 8 (Echo request)
  Code: 0
  Checksum: 0x4d2c (correct)
  Identifier: 0x0001
  Sequence number: 1
  Data: 32 bytes of arbitrary payload
```

### Key observations
- **RTT (Round Trip Time):** ~13–15ms to Google's servers
- **TTL (Time to Live):** 118 — Google's servers set TTL=128, each router hop decrements by 1. 10 hops were traversed.
- ICMP data payload is **not encrypted** — the payload content is visible
- Ping is useful for:
  - Checking if a host is alive
  - Measuring latency
  - Network path troubleshooting

### Security implication
- **ICMP Flood (Ping Flood):** Sending massive ICMP requests overwhelms the target
- **ICMP tunneling:** Attackers can exfiltrate data by hiding it inside ICMP payloads — completely invisible to firewalls that only filter by protocol/port
- Many organizations block ICMP at perimeter firewalls to prevent ping-based reconnaissance

## 5. Observation 4 – ARP Traffic

### Filter used
```
arp
```

### What was observed

```
Packet 1:  ARP  Who has 192.168.1.1? Tell 192.168.1.105     (Broadcast)
Packet 2:  ARP  192.168.1.1 is at AA:BB:CC:11:22:33          (Unicast reply)
Packet 3:  ARP  Who has 192.168.1.105? Tell 192.168.1.1      (Gratuitous ARP)
```

**ARP packet structure:**
```
Address Resolution Protocol (request)
  Hardware type: Ethernet (1)
  Protocol type: IPv4 (0x0800)
  Sender MAC: 11:22:33:44:55:66
  Sender IP:  192.168.1.105
  Target MAC: 00:00:00:00:00:00  (unknown — this is what we're asking for)
  Target IP:  192.168.1.1
```

### Key observations
- ARP is broadcast — **every device on the network sees every ARP request**
- ARP has **no authentication** — any device can claim to own any IP address
- Observed ARP cache on local machine:
```bash
# Windows command
arp -a

Interface: 192.168.1.105
  192.168.1.1    AA-BB-CC-11-22-33   dynamic   (router)
  192.168.1.255  FF-FF-FF-FF-FF-FF   static    (broadcast)
```

### Security implication — ARP Poisoning Detection
In an ARP poisoning attack, the attacker sends gratuitous ARP replies mapping the gateway IP to their own MAC:
```
Malicious ARP reply:
"192.168.1.1 is at 66:55:44:33:22:11" (attacker's MAC)
```

**How to detect:** Run `arp -a` and check if two different IPs share the same MAC address, or if the gateway IP maps to an unexpected MAC.

**Prevention:** Dynamic ARP Inspection (DAI) on managed switches validates ARP packets against a DHCP snooping binding table.

**Key insight:** The majority of traffic is encrypted HTTPS — demonstrating that modern web traffic is well-protected in transit. The remaining unencrypted protocols (DNS, ARP) are where attention should be focused for privacy improvements.

## 7. Key Takeaways from Traffic Analysis

1. **DNS reveals your browsing habits** — even when content is HTTPS-encrypted, domain names are visible in plaintext DNS queries. Use DoH or DoT for privacy.

2. **HTTPS works** — HTTPS/TLS traffic is genuinely opaque to a passive observer. Wireshark shows only `Application Data` with no content readable.

3. **ARP is fundamentally insecure** — no authentication means ARP poisoning is trivially easy on unprotected networks. Managed switches with DAI are essential.

4. **TLS SNI leaks the hostname** — even HTTPS reveals which domain you are connecting to via the SNI field in the TLS Client Hello.

5. **ICMP can be abused** — while useful for diagnostics, ICMP payloads are unencrypted and can carry hidden data (ICMP tunneling).

6. **Network traffic analysis is powerful** — even without decrypting content, metadata (who talks to whom, when, how much data) reveals a great deal about user behavior and system health.
